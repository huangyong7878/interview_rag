## 2. 如果要给客户试用

（请说明你会先和产品、客户或技术负责人确认哪些事情；你会如何理解目标、边界、验收标准和第一版交付范围。）

我会和产品经理沟通清楚需求的边界，比如是否使用外部vlm模型，用户文档的质量，类型，部署方式，使用限制等等。理解用户的最急迫的痛点后，根据第一性原理，共同建立一个第一版需求范围(尽量在第一版中解决用户的最急迫的痛点以加强后续客户的信心)：比如用户如果关心某些文档的质量导致识别不够，那么第一版中就需要体现这一点。交付范围必须体现用户最迫切的痛点，这是我认为的第一性原理

## 3. 如果要保证质量和稳定性

**验证可靠性：**

- 单元测试覆盖核心逻辑（分块、索引、意图分类、引用校验、声明提取），不依赖外部 API，秒级跑完
- 端到端测试覆盖关键场景：正文查询、表格查询、无答案拒答、跨条款合成，每条有明确的预期结果和接地分数阈值
- 每个修过的 bug 加回归测试保护——这次 BM25 分数错配、heading_path 拼接错误、LLM-as-judge 误判，修完都确认了对应测试通过
- 所有 LLM 调用打详细日志（`[query] Step 1/3`、`[indexer] About to instantiate...`），出问题时能快速定位到具体阶段

**排查流程（实际经历）：**

1. 先用 `/health` 确认索引状态和 chunk 数量，判断是索引问题还是查询问题
2. 看分步日志——这次 FlagEmbedding 崩溃，日志精确停在 `[indexer] About to instantiate BGEM3FlagModel...`，一秒定位到 torch segfault
3. 判断影响面，优先止损——`ENABLE_FAISS=false` 绕过崩溃，服务继续跑，根因后续排查

**止损策略：**

- 配置开关优先于代码修改（`ENABLE_FAISS`、`PARSER_PRIORITY`），几秒回滚，不用重新部署
- `force=true` 支持热重载索引，不丢线上状态

**修复确认：**

- 跑全量单元测试 + 端到端测试，确认修复不引入新问题
- 拿真实问题 case 重新查询，对比修复前后的 grounding_score、citation_errors、should_refuse

## 4. 如果时间有限

（请说明你会如何安排优先级、同步风险、协调资源，并保证关键目标尽量按时完成。）

根据第二条，也就是双方达成的第一性原理来决定优先级。比如如果用户最关心的是识别准确率，那么我会优先处理解析器的准确率问题；如果用户最关心的是查询效率，那么我会优先处理索引和检索的性能问题。

**同步风险**：我会把风险分为可控和不可控两类。可控风险（如代码 bug、性能瓶颈）通过测试和监控提前暴露；不可控风险（如外部 API 变更、第三方依赖故障）提前准备 fallback 方案。像这次 PaddleOCR 的 flat text 输出导致表格检索失效，虽然当前不是阻断性问题，但已经在代码里注释标注了风险和替代方案（切 VLM/MinerU）。风险一旦识别，第一时间同步给产品和直属领导，不等变成事故再报。

**协调资源**：如果任务依赖外部团队或第三方接口（比如这次如果要用 VLM API，涉及 API key 申请、预算审批），提前发起、给足缓冲，不等到开发阻塞再追。技术方案上尽量解耦——用 config 开关和 parser 策略链设计，让各个模块独立推进、互不阻塞。

**保证按时完成**：关键目标拆到以天为单位的可交付增量，每天有可演示的输出（哪怕是一个 endpoint 或一个测试用例）。如果中间发现某个任务耗时超出预期，立刻判断是"砍范围"还是"加人/换方案"，而不是默默加班拖到最后一天。

## 6. 补充一个真实例子

（请结合这次作业，或结合你过去的真实经历，讲一个你主动负责、推进问题闭环、在压力下保证结果的例子。重点说明你当时怎么判断、怎么沟通、怎么推进、最后结果如何。）

迪尔智融：AI 信贷审批数字化（3天→2小时）

  背景

  2022年约翰迪尔中国区农机信贷审批是全线下流程——经销商收纸质材料→发微信给信贷员→手工录入→人工查征信→逐级签字，平均3
  天，旺季压到5-7天。叠加疫情防控，经销商怨声载道，客户等着提车下地干活，审批不下来车发不走。这个项目当时不在全球 roadmap
  上，是我主动找中国区 MD 推动立项的。

  怎么判断

  我花两周画了完整 as-is 审批流程图，识别出四个瓶颈：资料采集慢（40%）、信息转录慢（25%）、征信查询慢（20%）、规则判
  断慢（15%）。核心判断是：真正拖慢审批的不是风控，而是信息收集和转录。机器做判断比人快且更一致。

  关键决策是不做一步到位，而是两阶段增量交付：第一阶段先用规则引擎+征信管道跑通自动预审（6周，3天→4小时），第二阶段
  再上 OCR+电子签章让经销商完全自助（6周，4小时→2小时）。每阶段都有独立的业务收益，万一第二阶段出问题，第一阶段的价
  值已经落袋。

  怎么沟通

  - 说服 CFO：用他自己的数据说话。
  - 获得风控团队支持：承诺"机审+人审分层"——规则引擎只处理明确通过和明确拒绝的单（约85%），中间灰色地带仍转人工。风控
  不是被替代，而是从重复劳动中解放出来聚焦复杂案例。额外承诺前三个月风控团队对系统审批有"一票事后否决权"——三个月零撤
  回后自动取消。给安全感，不挑战权威。
  - 经销商试点：选3家关系好、配合度高的经销商先跑，跑通后再推广，避免全面铺开出问题。

  怎么推进（压力点）

  央行征信接口联调比预期慢了3周，涉及资质审核、专线拉通、证书配置，商务和技术都不顺畅。我的做法：一是自己直接跟央行
  技术团队约会、逐条对测试报文，同时给商务团队画流程图让他们知道卡在谁手里；二是立刻启动 Plan
  B——先接第三方大数据公司的征信替代数据（工商司法、多头借贷等），让规则引擎第3周就产出预审结果，不等央行。

  结果

  - 审批时间：3天 → 2小时
  - 坏账率：维持原水平，未上升
  - 信贷员人效：日均5-8单 → 15-20单
  - 经销商满意度：60% → 95%+
  - 旺季审批积压从5-7天变为当天处理完毕
  - 项目后来成为全球 FinTech 团队参考案例，巴西团队做了类似改造

  复盘

  在业务压力下，最重要的不是方案有多完美，而是能不能让业务方在每个阶段都看到可感知的进展。信任是一点点攒出来的，技术
  、风控、业务三方的信任，各需要不同的沟通方式——数据、安全感、口碑。
